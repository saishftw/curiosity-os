# Curiosity OS

## UI / UX Vision

> **Design the experience around curiosity, not productivity.**

---

# Core Philosophy

Curiosity OS should not feel like a note-taking application, a project manager, or a productivity tool.

Instead, it should feel like a personal place that quietly preserves and grows curiosity over years.

The user should never open the application thinking:

* "What do I need to organize?"
* "What task should I complete?"

Instead, they should immediately think:

> "Oh... I forgot how interesting this was."

That emotional reaction is one of the primary design goals.

---

# The Mental Model

Do **not** think of Curiosity OS as another knowledge management system.

Think of it as:

> **A streaming service for your own mind.**

Streaming platforms don't ask users to organize content.

Instead, they gently invite users back into things they once loved.

Curiosity OS should do the same for ideas.

The UI should continuously encourage:

* Rediscovery
* Exploration
* Reflection
* Continuation

---

# Home

Home should **not** be a dashboard.

Avoid metrics such as:

* Number of ideas
* Active ideas
* Progress
* Completion
* Tasks

Instead, Home should feel like an invitation.

Example sections:

## Continue Thinking

The most relevant idea right now.

Display:

* Title
* Last insight
* Current intent

Primary action:

> Continue Thinking

---

## Rediscover

Ideas that once generated excitement but have been inactive for some time.

The purpose is to reignite curiosity.

---

## One Hour Explorations

Ideas that currently have a small, actionable next step.

Generated from the current Intent.

The user should immediately know:

> "I can meaningfully explore this in the time I have."

---

## Unexpected Connections

Surface meaningful relationships between ideas.

Do not simply show related ideas.

Explain **why** they are related.

Example:

Cybersecurity Training

↓

Behavior Design

↓

Learning Systems

Reason:

Both explore how human behavior changes through repeated interaction.

---

## Surprise Me

Randomly surface an interesting forgotten idea.

This should encourage serendipity.

---

# Browsing

Browsing should support multiple perspectives.

The repository should never have one rigid hierarchy.

Ideas should appear naturally across different views.

Examples:

## Emotional View

* 🔥 Became Interesting Again
* 🌱 New Curiosities
* 🚀 Ready to Build
* 🧠 Left Unfinished
* 🎲 Surprise Me

---

## Topic View

* AI
* Productivity
* Psychology
* Cybersecurity
* Learning
* Startups

---

## Activity View

* Exploring
* Building
* Researching
* Observing
* Paused

These are simply different lenses over the same repository.

# Sidebar

Desktop should use a lightweight contextual sidebar.

Example:

```
Home

Ideas

Connections

Search

----------------

(Current Idea)

Summary

Timeline

Knowledge

Resources

Sessions

Open Loops
```

The sidebar should change depending on context.

Avoid large navigation trees.

---

# Idea Page

Opening an Idea should **not** immediately reveal Markdown.

Instead, the user first receives a Resume Snapshot.

The page should answer four questions immediately.

1. What changed?
2. What did I learn?
3. Where did I stop?
4. What should I explore next?

Only after this should detailed documents become visible.

---

# Resume Snapshot

Suggested layout:

Current Summary

↓

Current Intent

↓

Last Insight

↓

Continue Thinking

Continue Building

Publish Session

This minimizes context rebuilding.

---

# Timeline

Timeline should read like the biography of an idea.

Avoid displaying raw commits or file changes.

Example:

```
Captured the idea.

↓

Discovered competitor.

↓

Changed business model.

↓

Prototype completed.

↓

Returned after three months.

↓

Discovered new direction.
```

Timeline tells the story.

Knowledge captures the understanding.

---

# Knowledge

Knowledge should feel like reading a beautifully written document.

Markdown should be rendered rather than edited by default.

Support:

* Images
* Code
* Diagrams
* Tables
* Callouts
* Embedded media

Reading should feel closer to documentation than note editing.

---

# Open Loops

Open Loops are not tasks.

They are invitations to continue thinking.

Examples:

* How do games teach habits?
* Can AI simulate phishing attacks?
* Explore BJ Fogg's behavioural model.

Never render them as checklists.

---

# Connections

Connections deserve their own visual treatment.

Do not present them as a simple graph or list.

Each connection should explain itself.

Example:

**Behavior Design**

Reason:

Referenced during cybersecurity research.

Shared concepts include:

* Habit formation
* Learning
* Gamification

The explanation builds trust.

---

# Search

Search should feel conversational.

Instead of

```
Search...
```

Prefer something like

> What are you thinking about?

Search should eventually support:

* Ideas
* Knowledge
* Resources
* Timeline
* Connections

---

# Capture

Capture must be frictionless.

Minimal example:

```
Title

Thought

Capture
```

Nothing else.

The user should not classify or organize during capture.

That responsibility belongs to the Librarian.

---

# Mobile

Mobile is the primary platform.

Suggested navigation:

```
Home

Ideas

Capture

Search

Profile
```

Capture should occupy the center position.

The application should be fully usable with one hand.

---

# Design Principles

Prefer:

* Whitespace
* Typography
* Readability
* Calm interfaces
* Progressive disclosure

Avoid:

* Dense dashboards
* Large tables
* Enterprise UI
* Excessive controls
* Complex configuration

---

# The Streaming Service Analogy

Borrow the interaction philosophy—not the visual style.

Examples:

Instead of:

* Recently Updated

Prefer:

* Continue Thinking

---

Instead of:

* Modified Yesterday

Prefer:

* Last Insight

---

Instead of:

* Recent Files

Prefer:

* Rediscover

---

Instead of:

* Related Items

Prefer:

* Unexpected Connections

---

The experience should gently pull users back into curiosity rather than asking them to manage information.

---

# The Librarian

The Librarian should become a subtle character within the product.

It is **not** a chatbot.

It is **not** an assistant that interrupts.

Instead, it quietly leaves thoughtful observations.

Examples:

* The Librarian found a connection between two ideas.
* The Librarian organized your latest exploration into three findings.
* The Librarian thinks this idea is ready to continue.
* The Librarian noticed a pattern across several ideas.

The Librarian should feel like someone quietly maintaining your personal library while you focus on thinking.

---

# Success Criteria

A successful interface should make the user feel:

* Curious
* Inspired
* Calm
* Excited to explore
* Able to immediately continue thinking

The user should visit Curiosity OS because they **want** to think—not because they **need** to manage their work.

---

# Guiding Question

Every UI decision should answer one question:

> **Does this make me want to open Curiosity OS even when I have nothing I need to do?**

If the answer is yes, it belongs.

If the answer is no, it probably belongs in a productivity application instead.
