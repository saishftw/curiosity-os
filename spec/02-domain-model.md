# Curiosity OS

## Document 2 — Domain Model & Entities

**Version:** v1.0

**Status:** Foundation

---

# Philosophy

Curiosity OS is fundamentally a knowledge system.

Everything in the application should revolve around a small number of well-defined entities.

The goal is to model how curiosity evolves rather than how projects are managed.

---

# Entity Hierarchy

```
Curiosity OS

└── Worlds
      ├── Knowledge
      ├── Thinking Sessions
      ├── Timeline
      ├── Portals
      ├── Context Packs
      ├── Connections
      ├── Open Loops
      ├── Resources
      ├── Derived Metadata
      └── Intent
```

The system should stay intentionally small.

Every feature should belong to one of these entities.

---

# 1. World

## Description

The primary entity of Curiosity OS.

A World represents a curiosity that evolves over time.

Examples:

* Startup ideas
* Research topics
* Personal decisions
* Learning goals
* Product concepts
* Technologies
* Creative ideas

A World is intentionally broader than a "project."

Projects eventually end.

Worlds continue evolving.

---

## World Properties

### Identity

* ID
* Name
* Slug

---

### Classification

A world may belong to multiple topics.

Examples

* AI
* Psychology
* Productivity
* Cybersecurity
* Startups
* Design

Topics are for browsing.

They are not the primary organization.

---

### Emotional Views

A world may appear in many emotional collections.

Examples

* 🔥 Became interesting again
* 🌱 New curiosity
* 🚀 Close to building
* 🧠 Left unfinished
* 🎲 Surprise me

These are derived rather than manually assigned.

---

### Summary

A concise overview describing the current understanding.

Generated by the Librarian.

---

### Current Intent

One sentence answering

> "If I have one free hour, what should I do?"

Unlike metadata, Intent is manually curated because it represents the user's current direction.

---

# 2. Knowledge

Knowledge is the permanent memory of a World.

Unlike sessions, Knowledge represents accumulated understanding.

Examples:

* Research
* Decisions
* Architecture
* Competitor analysis
* Technical findings
* Business models
* Market validation
* Interesting observations

Knowledge should be stored as Markdown.

---

## Knowledge Characteristics

Knowledge should be:

* Structured
* Readable
* Searchable
* AI-friendly
* Version controlled

Knowledge should evolve over time.

It should never be rewritten from scratch.

---

# 3. Thinking Session

The fundamental unit of work.

Everything meaningful begins as a Thinking Session.

Examples

* One hour in Claude
* Brainstorming in ChatGPT
* Coding session
* Reading papers
* Watching videos
* Whiteboarding
* Voice notes

Thinking Sessions are temporary.

Knowledge is permanent.

---

## Thinking Session contains

* Timestamp
* Portal
* Summary
* Insights
* Decisions
* Open Loops
* Connections
* Resources
* Intent Update

A Thinking Session is never edited after publication.

It becomes part of history.

---

# 4. Timeline

The Timeline records the evolution of a World.

Every Publish creates one Timeline Event.

Timeline Events are append-only.

History should never be rewritten.

---

Examples

```
Captured idea

↓

Validated market

↓

Discovered competitor

↓

Changed architecture

↓

Built prototype

↓

Paused

↓

Returned

↓

Found new direction
```

Timeline should be the fastest way to understand how an idea evolved.

---

# 5. Timeline Event

Each event represents one meaningful change.

Properties:

* Timestamp
* Portal
* Event Type
* Summary
* Linked Session
* Knowledge Delta

Events should be lightweight.

The Timeline is not a duplicate of Knowledge.

It is history.

---

# 6. Knowledge Delta

The output of every Publish.

Knowledge Deltas answer

"What changed?"

Examples

New knowledge

Invalidated knowledge

Questions answered

Questions created

Resources discovered

Connections discovered

Knowledge Deltas are applied by the Librarian.

---

# 7. The Librarian

The Librarian is the intelligence layer.

It is not a user-facing entity.

Responsibilities:

* Ingest Thinking Sessions
* Apply Knowledge Deltas
* Update Markdown
* Generate summaries
* Maintain structure
* Derive metadata
* Create timeline events
* Refresh Context Packs

The Librarian is the only component that understands repository structure.

---

# 8. Context Pack

A temporary package assembled for continuing work.

Context Packs are generated on demand.

They are not permanently stored.

---

A Context Pack may include

Current Summary

Recent Timeline

Important Decisions

Current Intent

Open Loops

Related Worlds

Key Knowledge

Relevant Resources

Recent Thinking Sessions

---

Purpose

Minimize context rebuilding.

---

# 9. Portal

A Portal is an external system where work happens.

Examples

* ChatGPT
* Claude
* GitHub Copilot
* VS Code
* GitHub
* Figma
* Apple Notes
* Voice Notes

Portals are temporary workspaces.

They are never considered the source of truth.

---

## Portal Responsibilities

Explore.

Research.

Build.

Publish results.

Nothing else.

---

# 10. Publish

Publish is not synchronization.

Publish creates a permanent Thinking Session.

Publishing should require minimal effort.

The result becomes part of Curiosity OS.

---

# 11. Connections

Connections describe relationships between Worlds.

Connections should emerge naturally.

Examples

Cybersecurity

↓

Psychology

↓

Behavior Design

↓

Habit Formation

↓

Education

↓

Gamification

Connections encourage cross-domain exploration.

---

Connections should support

* browsing
* recommendations
* related worlds
* curiosity graph

---

# 12. Open Loop

An unanswered question.

Examples

* Research competitor
* Validate pricing
* Explore API
* Prototype UX
* Read paper

Open Loops are not tasks.

They are curiosity waiting to be explored.

---

# 13. Resource

Anything worth referencing.

Examples

* Articles
* Videos
* Papers
* Images
* Websites
* Books
* GitHub repositories
* PDFs

Resources belong to Worlds.

Resources may also belong to Thinking Sessions.

---

# 14. Derived Metadata

Metadata should rarely be entered manually.

Instead, it should be inferred.

Examples

Momentum

Interest

Confidence

Novelty

Complexity

Activity

Evidence

Last Active

Session Count

Publish Count

Repository Activity

Research Activity

Open Loop Count

Connection Count

---

Metadata is for navigation.

Not documentation.

---

# 15. Momentum

Represents how actively a World is evolving.

Derived from

Recent publishes

Repository activity

Thinking Sessions

Timeline activity

---

# 16. Interest

Represents current excitement.

Influenced by

Frequency of visits

Publishing frequency

Exploration activity

Recent discoveries

---

# 17. Confidence

Represents certainty.

Confidence may decrease after learning.

Lower confidence is not failure.

It often means deeper understanding.

---

# 18. Evidence

Measures how much supporting knowledge exists.

Examples

Research

Prototypes

Market validation

User interviews

Architecture

Experiments

---

# 19. Novelty

Represents how unique or unexplored an idea currently feels.

Useful for discovery views.

---

# 20. Activity

Current dominant mode.

Derived from recent sessions.

Possible values

Exploring

Building

Observing

Paused

Researching

Activity should not be manually selected.

---

# 21. Intent

The only manually maintained state.

Purpose

Answer

"What should I do next?"

Examples

Research competitors.

Prototype onboarding.

Read three papers.

Validate assumptions.

Intent should always be actionable.

---

# Entity Relationships

```
World

├── owns Knowledge

├── owns Timeline

├── owns Thinking Sessions

├── owns Resources

├── owns Open Loops

├── owns Connections

├── owns Intent

├── owns Derived Metadata

└── generates Context Packs
```

---

# Design Rules

A World never directly edits Metadata.

Metadata is derived.

---

Thinking Sessions never modify Markdown.

They produce Knowledge Deltas.

---

The Librarian applies Knowledge Deltas.

---

Timeline is append-only.

---

Knowledge continuously evolves.

---

Context Packs are disposable.

---

Portals remain external.

---

Repository remains canonical.

---

# Future Extensions

The model should support future entities without changing the core architecture.

Possible future additions

* People
* Companies
* Books
* Technologies
* Locations
* Habits
* Conversations

These should integrate through Worlds rather than replacing them.

---

# Guiding Principle

Everything in Curiosity OS should answer one of four questions:

**Remember** — Preserve what I learned.

**Resume** — Help me continue.

**Connect** — Reveal relationships.

**Evolve** — Show how my thinking changed.

If a new entity does not strengthen one of these four capabilities, it probably does not belong in the system.
