# Curiosity OS

## Document 5 — Repository & Storage Specification

**Version:** v1.0

**Status:** Repository Specification

---

# Philosophy

The Git repository is the canonical memory of Curiosity OS.

It should be:

* Human readable
* AI friendly
* Git friendly
* Markdown first
* Portable
* Future proof

The repository is not optimized for databases.

It is optimized for thinking.

---

# Design Principles

The repository should:

* remain understandable without Curiosity OS
* work directly in GitHub
* work in VS Code
* work in Obsidian if desired
* support AI agents
* avoid proprietary formats

Everything important should remain plain text.


Bad


Good


"What do we know?"

# Repository Rules

Portals never edit Markdown directly.

↓

Thinking Sessions are immutable.

↓

Timeline is append-only.

↓

Knowledge is continuously curated.

↓

Metadata is always derived.

↓

Generated files are disposable.

↓

Repository remains human-readable.

---

# Git Philosophy

Commits represent

Thinking Sessions.

Not individual file edits.

One meaningful Publish

↓

One Git Commit

Commit messages should describe

knowledge gained

rather than

files modified.

Example

Good

"Discovered market positioning for cybersecurity training."

Bad

"Updated research.md"

---

# Naming Conventions

Use

lowercase

hyphen-separated

portable

human readable

Avoid spaces.

Avoid proprietary identifiers.

---

# Long-Term Compatibility

The repository should remain useful even if

Curiosity OS disappears.

Markdown readers should still work.

GitHub should still work.

VS Code should still work.

Obsidian should still work.

This repository is intended to last years.

# Open Questions

---

# Guiding Principle

If someone opens this repository directly in GitHub five years from now,

they should immediately understand

* what each idea is,
* what was learned,
* how thinking evolved,
* and where exploration should continue,

without needing Curiosity OS itself.
